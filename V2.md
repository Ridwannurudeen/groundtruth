## V2-1 (HEADLINE) — real semantic conflict detection, not DOI lookup

**Problem:** `contradictions.py:judge_retraction_contradiction` is `original_doi == claim_doi`. That only handles the trivial case (an external list hands you the exact DOI). The hard, interesting case — two remembered claims that *semantically* conflict with **no retraction notice** — is never attempted. That case is the only one that justifies a knowledge graph over a SQL `DELETE`.

**Build a two-tier detector** in `contradictions.py`:
- **Tier A (keep):** retraction-linked supersession via DOI match. Rename honestly (it's not "detection," it's a known-retraction lookup). This stays the deterministic baseline.
- **Tier B (new, the headline):** semantic conflict detection between claims with no retraction.
  1. For each claim, find candidate conflicting claims by **vector neighborhood** — embed the claim text and retrieve its nearest other claims (reuse the pattern in cognee's `coding_rule_associations.py`: vector `search` for endpoints), or `graph_engine.get_neighbors(node_id)` / `get_nodeset_subgraph(...)` (verified methods on the graph interface).
  2. For each candidate pair, an **LLM structured-output judge** (`LLMGateway.acreate_structured_output`, the real path you already use) returns `{conflicts: bool, direction: "a_supersedes_b"|"b_supersedes_a"|"mutual"|"none", basis: str, confidence: 0-1}`. Prompt it to distinguish genuine contradiction ("X reduces Y" vs "X has no effect on Y") from mere difference of topic. Gate on confidence ≥ 0.7; log rejected pairs.
  3. Emit a `contradicts` (and directional `supersedes`) edge for confirmed conflicts, stamped via the same `upsert_edges(..., data_id=...)` path already working.
- **Seed the corpus with real conflict pairs** so Tier B has something to find: add ~6–8 pairs of claims that genuinely conflict WITHOUT a retraction (e.g. an older study vs a newer meta-analysis reversing it). These are the cases that prove reasoning, not lookup.

**Gate V2-1:** on a held-out set of conflict pairs (with no retraction notice), Tier B flags the true conflicts and leaves unrelated pairs alone. Record precision/recall on that set in `docs/RESULTS-V2.md` with the actual pairs and verdicts. This is the number that matters now — not 0/20.

## V2-2 (HEADLINE) — put the graph in the answer loop

**Problem:** `answer.py` derives `cites_retracted` by token-matching `claims.json` + memory membership. It never reads recall output and never traverses the graph, so the graph and LLM are irrelevant to the score. The `contradicts` edge is never consulted.

**Rewrite the answer path to use the graph:**
1. Call `recall(..., include_references=True)` and take the **actual returned references** as the retrieved set (not a token re-match).
2. For each retrieved claim node, call `graph_engine.get_edges(node_id)` (verified) and check for incoming/outgoing `contradicts` / `supersedes` edges. If a retrieved claim is superseded, **surface it** ("this source was superseded by DOI …, per <basis>") and down-rank or exclude it from the answer.
3. Derive `cites_retracted` / `cites_superseded` from *this graph-aware pass over real recall output*. Keep the old `claims.json` check only as a secondary integrity assertion; if the two disagree, log it.
4. Return a **synthesized answer** (drop `only_context=True` or add a synthesis step) so the demo shows a real answer, not a node dump or template.

**Gate V2-2:** for 3 questions, `docs/RESULTS-V2.md` shows the recall references, the edges found per reference, and the resulting classification — demonstrating the decision came from the graph, not a JSON token match.

## V2-3 — supersede-with-provenance (don't throw away the story)

**Problem:** hard-`forget` deletes the node, destroying the history that makes the demo compelling, and the edge dies with it (never used).

**Add a retention mode:**
- On supersession, instead of immediately hard-forgetting, keep the original as a **tombstone**: retain the node, add a `superseded_by` edge to the retraction/superseding claim, and mark it (a `superseded: true` property, or move it to a `node_set="superseded"`). Recall (V2-2) then *knows* it's dead and annotates rather than silently dropping it — you can show "we believed X (2019) → superseded by Y (2024), here's the chain."
- Keep hard-`forget` as a distinct action (e.g. fraud/withdrawn-entirely, or a user "purge" button) so `forget` is still exercised for genuinely-dead nodes. Both paths demoable: *supersede* (retain history) vs *forget* (erase).
- This keeps the edge **alive and queried** (fixing the "edge is theater" critique) while preserving the surgical-forget win.

**Gate V2-3:** a query against a superseded claim returns the current answer AND the provenance chain (old claim → superseded_by → new). Hard-forget still works for a purge case. Both shown in `docs/RESULTS-V2.md`.

## V2-4 — real claims, not restated titles

**Problem:** `seed_corpus.json` claim text is the title with a prefix ("The paper claimed that <title>"), often ungrammatical — nothing concrete to contradict.

At curation (`ingest.py --curate`), have the LLM extract the paper's **actual finding as a concrete assertion** ("Compound Z reduces inflammation by ~30% in adults") from title + abstract/retraction reason, not a title restatement. Re-commit the regenerated `seed_corpus.json`. This makes both semantic detection (V2-1) and answers (V2-2) meaningful.

**Gate V2-4:** spot-check 5 claims read as real assertions, not title echoes.

## V2-5 — a thin agent loop (make "agent memory" true)

**Problem:** the pitch says "agent memory that unlearns," but it's a batch script + demo. No agent.

Add one small agent loop (`groundtruth/agent.py`): an assistant that answers user questions over a **persistent session**, using `recall` for memory. Then, when a retraction/supersession is ingested mid-session, the **same question asked again in a later turn returns a corrected answer** — live, not via a benchmark harness. This is the "memory changed the agent's behavior across sessions" moment judges reward, and it's a genuine use of Cognee session memory (`session_id`, `add_feedback`, `improve`). Keep it CLI-thin; wire it into the existing FastAPI demo as a "conversation" tab if time allows.

**Gate V2-5:** transcript in `docs/RESULTS-V2.md`: turn 1 answer → ingest retraction → turn N same question → corrected answer, same session.

Replace/augment the benchmark so it reports **two** numbers, clearly separated:
